## 一、大模型（LLM）配置

### 智谱 GLM（当前激活模型）

| 参数 | 值 |
|------|----|
| Provider | 智谱 AI (ZhipuAI) |
| API Base URL | `https://open.bigmodel.cn/api/paas/v4/` |
| 模型名称 | `glm-4.6`（在系统设置中配置） |
| API Key | 在系统"模型配置"页填写，**不硬编码** |
| 协议类型 | OpenAI 兼容（`/chat/completions`） |

### LLM 调用关键参数（必须设置）

```python
# Python (openai SDK) 调用示例
from openai import OpenAI

client = OpenAI(
    api_key="your-zhipuai-api-key",
    base_url="https://open.bigmodel.cn/api/paas/v4/"
)

completion = client.chat.completions.create(
    model="glm-4-flash",
    messages=[
        {"role": "user", "content": "..."}
    ],
    temperature=0.2,
    top_p=0.8,
    # ✅ 强制 JSON 输出（避免 markdown 包裹）
    response_format={"type": "json_object"},
    # ✅ 关闭 thinking 模式（加速响应，避免额外 token 消耗）
    extra_body={"thinking": {"type": "disabled"}}
)
```

```typescript
// TypeScript (前端 fetch 调用) 关键参数
const payload = {
    model: activeModel.name,
    messages: [...],
    temperature: 0.2,
    top_p: 0.8,
    response_format: { type: 'json_object' },  // ✅ 强制JSON
    thinking: { type: 'disabled' },             // ✅ 关闭thinking
};
```

> **重要说明：**
> - 智谱 API 的 `base_url` 末尾**必须**带 `/v4/`，系统会自动拼接 `/chat/completions`
> - `thinking` 参数在 GLM 系列中通过 `extra_body` 传入（Python SDK），直接写在 payload 里（TypeScript fetch）
> - 如果不关闭 thinking，响应时间会显著增加

## 二、PaddleOCR-VL 1.5

### 接口基础信息

| 参数 | 值 |
|------|----|
| API 地址（提交 Job） | `https://paddleocr.aistudio-app.com/api/v2/ocr/jobs` |
| API Token | `cf4a0b67edaf87887f0f5c1c81e4f5c89e9a4481` |
| 模型标识 | `PaddleOCR-VL-1.5` |
| 认证方式 | `Authorization: bearer <TOKEN>`（注意是小写 bearer） |

### 调用流程（异步 Job 模式）

```
1. POST /api/v2/ocr/jobs  →  提交 OCR 任务，返回 jobId
2. GET  /api/v2/ocr/jobs/{jobId}  →  轮询状态（pending/running/done/failed）
3. GET  {jsonUrl}  →  下载 JSONL 结果
```

### 请求示例

```python
import requests, json, time

JOB_URL = "https://paddleocr.aistudio-app.com/api/v2/ocr/jobs"
TOKEN   = "cf4a0b67edaf87887f0f5c1c81e4f5c89e9a4481"
MODEL   = "PaddleOCR-VL-1.5"

headers = {"Authorization": f"bearer {TOKEN}"}

optional_payload = {
    "useDocOrientationClassify": False,  # 关闭方向纠正（加速）
    "useDocUnwarping": False,            # 关闭扭曲纠正
    "useChartRecognition": False,        # 关闭图表识别
}

# ── 方式一：URL 模式（文件已在线上）─────────────────────────
headers["Content-Type"] = "application/json"
resp = requests.post(JOB_URL, headers=headers, json={
    "fileUrl": "https://...",
    "model": MODEL,
    "optionalPayload": optional_payload,
})

# ── 方式二：本地文件上传 ────────────────────────────────────
with open("file.pdf", "rb") as f:
    resp = requests.post(JOB_URL, headers=headers,
        data={"model": MODEL, "optionalPayload": json.dumps(optional_payload)},
        files={"file": f}
    )

job_id = resp.json()["data"]["jobId"]

# ── 轮询结果 ─────────────────────────────────────────────────
while True:
    r = requests.get(f"{JOB_URL}/{job_id}", headers=headers).json()
    state = r["data"]["state"]
    if state == "done":
        jsonl_url = r["data"]["resultUrl"]["jsonUrl"]
        break
    elif state == "failed":
        raise RuntimeError(r["data"]["errorMsg"])
    time.sleep(5)

# ── 解析结果 ─────────────────────────────────────────────────
lines = requests.get(jsonl_url).text.strip().split('\n')
for line in lines:
    page_result = json.loads(line)["result"]
    for layout in page_result["layoutParsingResults"]:
        # 纯文本（Markdown 格式）
        md_text = layout["markdown"]["text"]
        # 带 bbox 的块级结构
        for block in layout["prunedResult"]["parsing_res_list"]:
            print(block["block_label"], block["block_content"], block["block_bbox"])
```

### 返回 JSON 关键结构

```json
{
  "layoutParsingResults": [{
    "prunedResult": {
      "parsing_res_list": [
        {
          "block_label": "text",         // 块类型：text/header/paragraph_title/doc_title等
          "block_content": "正文内容",    // 文字内容
          "block_bbox": [x1, y1, x2, y2], // 矩形坐标（左上/右下）
          "block_id": 0,
          "block_order": 1,              // null 表示页眉页脚等非正文
          "block_polygon_points": [[x,y],...] // 精确多边形坐标
        }
      ]
    },
    "markdown": {
      "text": "# 标题\n\n正文..."  // 整页 Markdown 文本
    }
  }]
}
```

### 信息提取系统中的使用策略

| 文件类型 | 处理方式 | 说明 |
|---------|---------|------|
| PDF（扫描版）| ✅ 走 OCR | 提交文件 URL，获取 `parsing_res_list` |
| 图片（jpg/png 等）| ✅ 走 OCR | 同上 |
| PDF（可编辑版）| ⚡ 直接解析 | 未来可用 pymupdf 提取，无需 OCR |
| DOCX | ⚡ 直接解析 | python-docx 提取段落文本 |
| TXT | ⚡ 直接解析 | 直接 decode |

> **与 LLM 结合的源码匹配：** OCR 的 `block_content` 即为 LLM 的 source 字段的匹配基础。
> LLM 返回的 `source` 字段需在 `parsing_res_list` 各 `block_content` 中做字符串匹配，匹配后可取 `block_bbox` 用于前端高亮绘制。

## 三、Supabase

| 参数 | 获取位置 |
|------|---------|
| Project URL | Supabase 控制台 → Settings → API → Project URL |
| Anon Key（前端用） | Settings → API → `anon public` |
| Service Role Key（后端用） | Settings → API → `service_role secret`（⚠️ 勿暴露） |
| Storage Bucket 名称 | `files`（私有，RLS 隔离） |

**前端**（Next.js）使用 `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY`  
**后端**（FastAPI）使用 `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`（绕过 RLS）

## 五、重要变量名速查

| 变量名 | 含义 | 作用域 |
|--------|------|--------|
| `PADDLEOCR_TOKEN` | OCR API 鉴权 Token | 后端 |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase 服务端密钥 | 后端 |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase 前端公钥 | 前端 |
| `is_active` | 模型配置表字段，全局只一个生效 | DB |
| `is_default` | 提示词组表字段，用户默认词组 | DB |
| `prompt_set_id` | Task 关联的提示词组 | DB |
| `model_config_id` | Task 关联的模型配置 | DB |
| `extraction_results.source` | LLM 返回的原文出处（与 OCR block_content 匹配） | DB |
| `extraction_results.bbox` | 匹配到的文字块坐标（JSONB） | DB |
